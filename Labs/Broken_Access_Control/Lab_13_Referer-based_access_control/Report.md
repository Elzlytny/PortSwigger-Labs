# BAC-013 - Referer-based access control

## Report Information

- **Category:** Broken Access Control
- **Subcategory:** Vertical Privilege Escalation
- **Severity:** High

---

## Executive Summary

The application relies on the client-controlled `Referer` header when authorizing administrative actions.

A legitimate administrative request was captured and replayed after replacing only the administrator's session cookie with a regular user's session.

Although the request originated from a low-privileged user, the server accepted it because it incorrectly relied on the preserved `Referer` header instead of validating the authenticated user's permissions.

As a result, a regular user can perform administrative actions and escalate privileges.

---

## Affected Components

- Administrative role management
- Authorization mechanism
- HTTP request validation

---

## Vulnerability Description

The application uses the client-supplied `Referer` header as part of its authorization mechanism.

Because this header is entirely controlled by the client, it cannot be trusted for security decisions.

A legitimate administrative request can be replayed using a low-privileged user's session while preserving the original `Referer` header.

Since the server does not properly validate the authenticated user's privileges, it executes the privileged action successfully.

This issue results in a **Vertical Privilege Escalation** vulnerability.

## Impact

Successful exploitation could allow an attacker to:

- Escalate privileges.
- Perform unauthorized administrative actions.
- Bypass authorization controls.
- Compromise the integrity of the application's authorization model.

---

## Root Cause

The application incorrectly trusts the client-controlled `Referer` header when making authorization decisions.

Instead of validating the authenticated user's privileges on the server side for every privileged request, the application partially relies on request metadata that can be captured and replayed by an attacker.

---

## Remediation

To prevent this issue:

- Never rely on the `Referer` header for authorization decisions.
- Perform authorization checks exclusively on the server side.
- Validate user privileges for every privileged request.
- Treat all client-supplied headers as untrusted input.
- Regularly test access control mechanisms during security assessments.
