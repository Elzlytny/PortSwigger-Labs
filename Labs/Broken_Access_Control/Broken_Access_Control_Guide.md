# 🔐 Broken Access Control

تخيل إنك فاتح تطبيق بنك. 🏦

بعد ما سجلت دخولك، قدرت تشوف حسابك، تحول فلوس، تراجع معاملاتك، وتتابع بطاقاتك البنكية.

لكن...

إيه اللي هيحصل لو حاولت:

- 👤 تشوف حساب عميل تاني؟
- 👑 تدخل لوحة تحكم الأدمن؟
- 💰 توافق على قرض مع إنك مستخدم عادي؟
- 🛒 تعدل على طلب كنت دفعت تمنه بالفعل؟
- 📄 تنزل كشف حساب يخص مستخدم تاني؟

أكيد التطبيق المفروض يمنع كل المحاولات دي.

ولو سمح بأي واحدة منهم...

فإحنا غالبًا قدام **Broken Access Control**.

لكن قبل ما نفهم يعني إيه Broken Access Control...

لازم نسأل سؤال أهم.

---

# 🤔 هو ليه Access Control موجود أصلاً؟

أي Web Application في الدنيا بيحتوي على بيانات.

ودي ممكن تكون:

- بيانات مستخدمين.
- صور.
- ملفات.
- طلبات.
- رسائل.
- إعدادات.
- وظائف إدارية.
- APIs.
- أو حتى مجرد زرار المفروض الأدمن بس هو اللي يشوفه.

مش كل المستخدمين المفروض يشوفوا نفس البيانات.

ومش كلهم المفروض يقدروا ينفذوا نفس العمليات.

وده معناه إن التطبيق لازم يكون عنده طريقة يحدد بيها:

> مين يقدر يعمل إيه؟

ودي هي وظيفة **Access Control**.

ببساطة...

هو النظام المسؤول عن تحديد إذا كان المستخدم الحالي مسموح له يوصل للـ Resource أو ينفذ الـ Action اللي طلبها ولا لا.

---

# 🧠 السيرفر بيفكر إزاي؟

ناس كتير بتتعامل مع الـ Request على إنها مجرد Request رايح للسيرفر وخلاص.

لكن الحقيقة...

كل Request بيوصل للسيرفر بيعدي على سلسلة من الأسئلة.

خلينا نتخيل إن المستخدم بعت الـ Request دي:

```http
GET /admin
```

قبل ما السيرفر ينفذ أي حاجة...

بيبدأ يسأل نفسه:

---

## السؤال الأول...

> مين اللي بعت الـ Request دي؟

وده اسمه:

## 🔑 Authentication

يعني:

هل المستخدم ده مسجل دخول أصلاً؟

ولو آه...

مين هو؟

هل هو:

- Ahmed؟
- Carlos؟
- Wiener؟
- Administrator؟

لو معرفتش هو مين...

فالـ Request بتترفض غالبًا.

---

## السؤال التاني...

طيب...

بعد ما عرفت مين المستخدم...

السؤال اللي بعده بيكون:

> هو المستخدم ده المفروض يعمل إيه؟

وده اسمه:

## 🛡️ Authorization

يعني:

إيه الصلاحيات اللي عنده؟

هل هو:

- User؟
- Moderator؟
- Admin؟

كل Role ليها صلاحيات مختلفة.

---

## السؤال التالت...

بعد ما عرفت شخصيته...

وعرفت صلاحياته...

بييجي أهم سؤال.

> هل العملية اللي طلبها دي ضمن صلاحياته فعلًا؟

وده هو:

## 🚦 Access Control

يعني...

لو المستخدم User...

وبعت:

```http
GET /admin
```

هل المفروض يدخل؟

لا.

ولو Admin...

يبقى آه.

هنا السيرفر بيطبق قواعد الـ Access Control.

---

## السؤال الرابع...

طب لو حد سرق Session قديمة؟

أو بعت Request قديمة؟

أو حاول يستخدم Session انتهت؟

السيرفر لازم يتأكد إن الـ Request لسه جاية من Session صحيحة.

وده بيقع تحت:

## 🍪 Session Management

ودي مسؤولة عن:

- إنشاء الـ Session.
- تخزينها.
- التحقق منها.
- إنهائها عند Logout أو انتهاء الوقت.

---

# 💡 الفرق بينهم

ناس كتير جدًا بتخلط بين الأربع مصطلحات دول.

خلينا نبسطهم جدًا.

**Authentication**

يجاوب على سؤال:

> مين أنت؟

---

**Authorization**

يجاوب على سؤال:

> إيه الصلاحيات اللي عندك؟

---

**Access Control**

يجاوب على سؤال:

> هل العملية اللي بتحاول تعملها مسموحلك تعملها فعلًا؟

---

**Session Management**

يجاوب على سؤال:

> هل الـ Request دي فعلًا جاية من المستخدم اللي سجل دخوله؟

---

# 🧩 طيب هو ليه فصلنا بينهم؟

لأن كل واحد منهم ممكن يكون فيه ثغرات مستقلة.

مثلًا...

لو قدرت تدخل بحساب شخص تاني بدون باسورد...

فدي غالبًا مشكلة في:

**Authentication**

---

ولو دخلت بحسابك العادي...

لكن قدرت تدخل لوحة الأدمن...

فدي مشكلة في:

**Access Control**

---

ولو سرقت Session مستخدم تاني...

واستخدمتها...

فدي غالبًا مشكلة في:

**Session Management**

---

علشان كده...

رغم إن الأربع حاجات بيكملوا بعض...

لكن كل واحدة تعتبر جزء مستقل من أمن التطبيق.

---

# 🚨 إمتى نقول إن عندنا Broken Access Control؟

ببساطة...

لما التطبيق يفشل في التأكد إن المستخدم الحالي يملك الصلاحية للوصول إلى Resource أو تنفيذ Functionality معينة.

لاحظ كلمة:

> **يفشل في التأكد**

مش شرط يكون مفيش Authentication.

ومش شرط يكون المستخدم غير مسجل دخول.

بالعكس...

أغلب ثغرات Broken Access Control بتحصل والمستخدم بالفعل مسجل دخول.

المشكلة إن التطبيق بيطبق قواعد الصلاحيات بشكل خاطئ.

وده ممكن يسمح للمهاجم إنه:

- يشوف بيانات مستخدمين تانيين.
- يعدل بياناتهم.
- ينفذ عمليات إدارية.
- يحصل على معلومات حساسة.
- أو حتى يسيطر على التطبيق بالكامل.

---

# 🧠 قبل ما نكمل...

عاوزك من دلوقتي تغير طريقة التفكير.

متبصش لأي Request وتسأل:

> هل دي فيها ثغرة؟

اسأل السؤال ده بدلها:

> المفروض مين بس اللي يقدر ينفذ الـ Request دي؟

بعدها اسأل:

> طيب...

> إيه اللي هيمنعني أجرب؟

لو بدأت تفكر بالطريقة دي...

هتكتشف إن أغلب ثغرات Broken Access Control بتبدأ من مجرد سؤال بسيط جدًا...

> **"هو التطبيق متأكد فعلًا إني أملك الصلاحية؟"**

وده السؤال اللي هنبني عليه كل اللي جاي.

# 🛡️ Types of Access Control

دلوقتي عرفنا يعني إيه Access Control.

لكن...

هل كل مشاكل الـ Access Control شبه بعض؟

الإجابة لا.

علشان كده الناس قسمت Access Control إلى 3 أنواع رئيسية.

ليه؟

لأن كل نوع بيحمي حاجة مختلفة.

ولو فهمت الفرق بينهم...

هتقدر تحدد نوع الثغرة من أول كام دقيقة أثناء الاختبار.

الأنواع هم:

- ⬆️ Vertical Access Control
- ↔️ Horizontal Access Control
- 🔄 Context-dependent Access Control

خلينا نفهم كل واحد منهم من البداية.

---

# ⬆️ Vertical Access Control

أول نوع هو أسهل واحد نفهمه.

خلينا نتخيل إن عندنا تطبيق لإدارة شركة.

وفيه 3 أدوار.

```text
Administrator
      ▲
Manager
      ▲
Employee
```

كل ما تطلع لفوق...

صلاحياتك بتزيد.

يعني الأدمن يقدر يعمل كل حاجة.

المانجر يقدر يعمل حاجات أقل.

الموظف العادي يقدر يعمل أقل منهم.

لاحظ إنهم مرسومين فوق بعض.

ومن هنا جه اسم:

> **Vertical**

لأن الفرق بينهم فرق في **مستوى الصلاحيات**.

---

طيب...

لو أنا Employee...

وحاولت أدخل:

```text
/admin
```

أو أحذف مستخدم.

أو أضيف مستخدم.

أو أغير Role.

أو أشوف لوحة التحكم.

يبقى أنا بحاول أطلع لفوق.

يعني بحاول أوصل لصلاحيات أعلى مني.

وده اسمه:

> **Vertical Privilege Escalation**

---

## 💡 مثال

أنا User عادي.

لكن بعت Request بالشكل ده:

```http
POST /admin/deleteUser
```

ولو السيرفر نفذه...

يبقى التطبيق سمحلي أستخدم Functionality المفروض خاصة بالأدمن.

ودي تعتبر Broken Access Control.

---

## طيب...

السؤال المهم هنا.

إيه اللي التطبيق المفروض يحميه؟

الإجابة:

**Functionality**

يعني العمليات نفسها.

مش البيانات.

الهدف هنا غالبًا بيكون:

- دخول Admin Panel.
- حذف مستخدم.
- تعديل صلاحيات.
- إدارة النظام.
- تغيير إعدادات.
- إنشاء مستخدمين.

كل دي Functionality.

---

## 🧠 فكر كده...

لما تلاقي أي زرار.

اسأل نفسك.

> مين المفروض يقدر يستخدمه؟

لو الإجابة:

> الأدمن بس.

يبقى خلاص.

لقيت نقطة محتملة لاختبار Vertical Access Control.

---

# أشهر صور Vertical Access Control

بعد ما فهمنا الفكرة...

تعالى نشوف التطبيقات بتغلط فيها إزاي.

هتلاحظ إن كل السيناريوهات اللي جاية...

رغم إنها مختلفة.

لكن كلها في الآخر بتحاول تجاوب على نفس السؤال.

> إزاي مستخدم عادي وصل لحاجة المفروض الأدمن بس هو اللي يوصلها؟

---

## 🚪 1. Unprotected Administrative Functionality

وده أبسط سيناريو.

المبرمج يعمل Admin Panel...

وينسى يحط عليها Authorization أصلاً.

فتكون الصفحة موجودة عادي.

وأول ما تعرف الرابط...

تدخل.

زي:

```text
/admin
```

أو

```text
/administrator
```

أو

```text
/secret-admin
```

---

## طب لو الرابط مش معروف؟

هنا يبدأ الـ Recon.

ممكن تلاقيه في:

- robots.txt
- sitemap.xml
- HTML Source
- JavaScript
- Burp Sitemap

ولو ملقيتوش.

نبدأ Content Discovery.

زي FFUF.

---

### الفكرة هنا

الثغرة مش إنك عرفت الرابط.

الثغرة إن السيرفر سمحلك تستخدمه.

---

> ✅ Key Idea
>
> معرفة الـ Endpoint لا تعني امتلاك صلاحية استخدامه.

---

## 🍪 2. Client-Controlled Authorization

السيناريو اللي بعده أخطر شوية.

المبرمج بيعمل Authorization.

لكن...

بيعتمد على بيانات جاية من المستخدم نفسه.

يعني بدل ما السيرفر يحتفظ بالمعلومة...

يسيب المستخدم هو اللي يقولها.

مثلاً.

يبعت Cookie بالشكل ده.

```http
Cookie:
admin=false
```

طيب...

مين اللي بعت الـ Cookie؟

المستخدم.

يبقى...

مين يقدر يعدلها؟

المستخدم.

فلو غيرناها إلى:

```http
Cookie:
admin=true
```

ودخلنا Admin Panel.

يبقى المشكلة فين؟

المشكلة إن التطبيق وثق في بيانات المستخدم.

---

ونفس الفكرة مع:

- URL Parameters

```text
role=user
```

↓

```text
role=admin
```

---

أو Body Parameters.

```json
{
    "roleId":1
}
```

↓

```json
{
    "roleId":2
}
```

---

أو Hidden Fields.

---

أو حتى Headers.

---

لاحظ إن كل الأمثلة دي مختلفة.

لكن كلها راجعة لفكرة واحدة.

> التطبيق بيصدق أي حاجة المستخدم بيقولها.

وده أخطر خطأ ممكن يحصل في Authorization.

---

> ✅ Key Idea
>
> أي بيانات جاية من الـ Client تعتبر User-Controlled Data.
>
> وبالتالي لا يجوز الاعتماد عليها لاتخاذ قرار أمني.

---

## قبل ما نكمل...

ممكن تكون لاحظت حاجة.

كل السيناريوهات اللي فاتت...

كانت بتحاول توصل لحاجات تخص Role أعلى.

يعني كلها كانت:

User

↓

Admin

لكن...

إيه اللي يحصل لو أنا مش عايز أوصل للأدمن أصلاً؟

أنا عايز أوصل لمستخدم تاني...

نفس دوري بالظبط.

هنا بنخرج من Vertical Access Control...

وندخل على النوع التاني.

**Horizontal Access Control.**

# ↔️ Horizontal Access Control

وصلنا دلوقتي للنوع التاني.

وده النوع اللي بيلخبط ناس كتير.

خصوصًا أول مرة يدرسوا Broken Access Control.

تعالى الأول نرجع خطوة لورا.

في الـ Vertical...

أنا كنت بحاول أوصل لصلاحيات أعلى مني.

يعني:

```text
User
   ↓
Admin
```

لكن هنا...

أنا مش عايز أبقى Admin.

أنا عايز أوصل لحاجة تخص مستخدم تاني...

بس هو نفس دوري بالظبط.

يعني:

```text
User A  ←→  User B
```

أو

```text
Admin A  ←→  Admin B
```

أو

```text
Employee A  ←→  Employee B
```

كلهم نفس الـ Role.

لكن كل واحد ليه بياناته الخاصة.

وليه العمليات الخاصة بيه.

---

خلينا نسأل سؤال.

لو أنا User.

هل المفروض أشوف Profile بتاع User تاني؟

لا.

---

هل المفروض أعدل بياناته؟

لا.

---

هل المفروض أنزل ملفاته؟

لا.

---

هل المفروض أغير باسورده؟

أكيد لا.

---

يبقى رغم إننا الاتنين Users...

لسه فيه Access Control.

وده هو:

**Horizontal Access Control.**

---

# 💡 طيب ليه اسمه Horizontal؟

بص على الرسمة دي.

```text
User A  ←→  User B
```

إحنا مش طالعين لفوق.

ومش نازلين لتحت.

إحنا بنتحرك يمين وشمال.

بين مستخدمين في نفس المستوى.

علشان كده اسمه:

> Horizontal

---

# 🧠 طيب التطبيق هنا بيحمي إيه؟

ودي نقطة مهمة جدًا.

في Vertical...

التطبيق كان بيحمي:

**Functionality**

زي:

- Admin Panel
- Delete User
- Create User

لكن هنا...

غالبًا بيحمي:

**Resources**

زي:

- الحساب.
- الصور.
- الملفات.
- الطلبات.
- الرسائل.
- الفواتير.
- المحادثات.
- الـ API Keys.

لكن...

ممكن يحمي Functionality برضو.

زي إنك تقدر تغير Email مستخدم تاني.

أو تمسح Order يخص مستخدم تاني.

يبقى الـ Horizontal بيحمي:

- Resources
- Functionality

طالما هما يخصوا مستخدم آخر بنفس مستوى الصلاحيات.

---

# 🤔 طيب...

هو إزاي أصل للـ Resources دي؟

وهنا بقى بنوصل لأهم نقطة في الفصل كله.

ناس كتير فاكرة إن:

Horizontal Access Control

=

IDOR

لكن...

دي مش الحقيقة.

---

# ⚠️ أهم نقطة في الملف كله

**Horizontal Access Control هو نوع من أنواع Broken Access Control.**

أما

**IDOR فهو مجرد طريقة من طرق استغلال Horizontal Access Control.**

يعني العلاقة بينهم بالشكل ده.

```text
Broken Access Control
│
├── Vertical Access Control
│
├── Horizontal Access Control
│      │
│      ├── IDOR
│      ├── Session-based Issues
│      ├── Business Logic Flaws
│      └── ...
│
└── Context-dependent Access Control
```

يبقى.

Horizontal هو:

**Category**

أما

IDOR فهو:

**Scenario**

ودي نقطة مهمة جدًا.

---

# 🆔 طيب يعني إيه IDOR؟

IDOR اختصار لـ:

> **Insecure Direct Object Reference**

الاسم شكله كبير.

لكن لما نفهمه كلمة كلمة...

هنكتشف إنه بسيط جدًا.

---

## Insecure

يعني غير آمن.

---

## Direct

يعني أنا بوصل للـ Object مباشرة.

من غير وسيط.

---

## Object

ودي أهم كلمة.

إيه هو الـ Object؟

أي حاجة التطبيق بيتعامل معاها.

زي:

- User
- Order
- Invoice
- File
- Message
- Image
- API Key

كل واحدة من دول تعتبر Object.

---

## Reference

يعني الطريقة اللي التطبيق بيستخدمها علشان يشاور على الـ Object.

ممكن تكون:

```text
?id=15
```

أو

```text
?id=carlos
```

أو

```text
?uuid=84d9...
```

أو

```text
/chat/2.txt
```

كل دي References.

---

# 💡 يبقى الفكرة كلها إيه؟

التطبيق بيقول.

"هاتلي الـ ID...

وأنا هرجعلك الـ Object."

وده طبيعي جدًا.

المشكلة إمتى؟

لما ينسى يتأكد...

هل المستخدم أصلًا من حقه يشوف الـ Object ده؟

---

# مثال

أنا فاتح حسابي.

اللينك كده.

```text
/my-account?id=wiener
```

طيب...

مين قال إن السيرفر لازم يرجع Account Wiener؟

هو رجعه...

علشان الـ ID بتاعه Wiener.

طيب...

نجرب نغيره.

```text
/my-account?id=carlos
```

لو السيرفر رجع بيانات Carlos...

يبقى المشكلة حصلت.

---

لاحظ...

أنا معملتش SQL Injection.

ولا XSS.

ولا RCE.

أنا غيرت كلمة واحدة بس.

ومع ذلك...

وصلت لبيانات مستخدم تاني.

وده هو:

**IDOR**

---

# 🧠 السؤال المهم

هل كل IDOR تعتبر Horizontal؟

في أغلب الحالات...

آه.

لأنك غالبًا بتوصل لمستخدم تاني.

لكن...

ممكن تبدأ Horizontal...

وتتحول بعد كده إلى Vertical.

إزاي؟

وده اللي هنشوفه في الجزء اللي جاي.

# 🧩 أنواع الـ IDOR

عرفنا دلوقتي إن IDOR بيحصل لما التطبيق يسمح للمستخدم بالوصول إلى Object عن طريق Identifier يقدر يتحكم فيه، من غير ما يتأكد إنه فعلًا يملك صلاحية الوصول للـ Object ده.

لكن...

هل كل ثغرات IDOR شبه بعض؟

الإجابة لا.

الفكرة الأساسية واحدة.

لكن تأثيرها بيختلف حسب نوع البيانات اللي التطبيق بيرجعها.

# 📄 1. Basic IDOR

وده أبسط شكل ممكن تقابله.

نفترض إن التطبيق بيعرض صفحة الحساب بالشكل ده.

```text
/my-account?id=wiener
```

المستخدم الطبيعي هيشوف حسابه هو.

طيب...

إيه اللي يمنعني أجرب؟

```text
/my-account?id=carlos
```

لو التطبيق رجع Account Carlos...

يبقى عندنا IDOR.

لاحظ إن اللي حصل بسيط جدًا.

أنا غيرت Identifier فقط.

والتطبيق رجع Object يخص مستخدم تاني.

---

> ✅ Key Idea
>
> لو مجرد تغيير الـ Identifier سمحلك توصل لـ Resource تخص مستخدم آخر، فدي تعتبر IDOR.

---

# 🔑 2. IDOR with Sensitive Information Disclosure

أحيانًا...

التطبيق مش بيرجع Profile كامل.

لكن بيرجع معلومة حساسة جدًا.

زي:

- API Key
- Email
- Phone Number
- Access Token
- Secret Value

خلينا نفترض إن الصفحة دي.

```text
/my-account?id=wiener
```

بتعرض:

```text
API Key:
ABCD1234
```

طيب...

غيرنا الـ ID.

```text
/my-account?id=carlos
```

وفجأة...

رجع API Key الخاصة بـ Carlos.

هنا برضو السبب هو IDOR.

لكن الـ Impact أكبر.

لأن البيانات اللي اتسربت تعتبر Sensitive Information.

وده بالظبط اللي حصل في أحد لابات PortSwigger.

---

> ✅ Key Idea
>
> المشكلة الأساسية مازالت IDOR.
>
> لكن نوع البيانات اللي رجعت هو اللي زود خطورة الثغرة.

---

# 🔐 3. IDOR with Password Disclosure

وده من أخطر السيناريوهات.

الفكرة نفسها.

غيرت الـ Identifier.

لكن بدل ما التطبيق يرجع بيانات عادية...

رجع Password.

مثلاً.

```text
/my-account?id=administrator
```

ورجع:

```text
Password:
admin123
```

في الحالة دي...

أنا لسه مستغلتش الثغرة.

أنا لسه جمعت معلومات.

بعدها أستخدم الباسورد.

وأدخل كـ Administrator.

يبقى التسلسل كان كده.

```text
IDOR
      │
      ▼
Password Disclosure
      │
      ▼
Login as Administrator
      │
      ▼
Vertical Privilege Escalation
```

يبقى...

الثغرة بدأت Horizontal.

وانتهت Vertical.

---

> ✅ Key Idea
>
> Password Disclosure مش بديل لـ IDOR.
>
> ده مجرد أثر (Impact) ناتج عن استغلال IDOR.

---

# 🔀 4. IDOR with Information Disclosure During Redirect

أحيانًا...

تغير الـ Identifier.

فتلاقي السيرفر رد:

```http
302 Found
```

أول حاجة ممكن تيجي في دماغك.

"تمام...

الموقع عمل Redirect.

يبقى المحاولة فشلت."

لكن...

استنى.

هل بصيت على الـ Response نفسه؟

ولا بصيت على Status Code بس؟

بعض التطبيقات بتبني الـ Response بالكامل.

وبعدين تعمل Redirect.

يعني ممكن يحصل كده.

```http
302 Found
```

لكن جوه الـ Response Body.

تلاقي بيانات Carlos.

رغم إن المتصفح هيعمل Redirect بعد ثواني.

وده معناه إن البيانات بالفعل خرجت من السيرفر.

وده يعتبر Information Disclosure.

لكن...

إيه السبب الأساسي؟

لسه هو IDOR.

لأن البيانات ظهرت بعد تغيير الـ Identifier.

---

> ✅ Key Idea
>
> متخليش Status Code يخدعك.
>
> راجع الـ Response بالكامل قبل ما تعتبر المحاولة فشلت.

# ⬆️ Horizontal to Vertical Privilege Escalation

ودي من أكتر الحاجات اللي بتتكرر في الواقع.

أنت في الأول...

مكنتش بتحاول تبقى Admin.

أنت كنت بتحاول توصل لمستخدم تاني.

لكن...

المستخدم التاني طلع Administrator.

أو قدرت توصل لـ:

- Password
- Session
- API Key
- Reset Token

بعدها استخدمت البيانات دي.

وفجأة.

بقيت Admin.

يبقى.

الثغرة بدأت.

```text
Horizontal Access Control
```

ثم.

```text
IDOR
```

ثم.

```text
Sensitive Information Disclosure
```

ثم.

```text
Vertical Privilege Escalation
```

وده مهم جدًا.

لأن ناس كتير أول ما تدخل Admin.

تقول:

"دي Vertical."

لا.

هي انتهت Vertical.

لكن بدأت Horizontal.

وده بيفرق جدًا أثناء كتابة الـ Report.

# 🔄 Context-dependent Access Control

بعد ما خلصنا Vertical و Horizontal...

وصلنا لآخر نوع من أنواع Access Control.

وده في رأيي من أكتر الأنواع اللي بتتعب البنتستر في البداية.

ليه؟

لأنك ساعات تبقى شايف كل حاجة متأمنة.

والـ Authorization شغال.

ومفيش أي Parameter ينفع يتغير.

ومفيش IDOR.

ولا Cookies.

ولا Headers.

ومع ذلك...

الثغرة موجودة.

إزاي؟

علشان المشكلة هنا مش في الـ User.

ولا في الـ Resource.

المشكلة في...

**ترتيب تنفيذ الخطوات.**

---

خلينا ناخد مثال من حياتنا.

أنت رايح البنك علشان تسحب مبلغ كبير.

الموظف بيعمل إيه؟

1. يتأكد من هويتك.
2. يتأكد إن الرصيد يكفي.
3. يراجع الطلب.
4. يطلب منك تأكيد.
5. يصرف الفلوس.

الخطوات دي لازم تحصل بنفس الترتيب.

مينفعش تبدأ من رقم 5.

---

ونفس الفكرة بتحصل في مواقع الويب.

في عمليات كتير بتتكون من أكتر من Request.

زي:

- تغيير صلاحيات مستخدم.
- تحويل فلوس.
- حذف حساب.
- تغيير كلمة المرور.
- شراء منتج.
- تأكيد طلب.
- الموافقة على قرض.

كل دي اسمها:

**Multi-step Processes**

# 📨 Referer-based Access Control

دلوقتي هنتكلم عن سيناريو بيحصل قليل.

لكن مهم جدًا.

خصوصًا لما تشتغل Bug Bounty.

خلينا الأول نفهم.

يعني إيه Referer؟

لما تكون فاتح صفحة.

وروحت لصفحة تانية.

الـ Browser ساعات بيضيف Header اسمه.

```http
Referer
```

ويقول للسيرفر.

"أنا جاي من الصفحة دي."

مثلاً.

```http
Referer:
/admin
```

---

المفروض...

السيرفر يعمل إيه؟

ولا حاجة.

يستخدم الـ Referer للإحصائيات.

أو الـ Analytics.

أو تسجيل الأحداث.

لكن...

ميستخدموش في الأمن.

---

# للأسف...

بعض التطبيقات بتعمل كده.

تقول.

"طالما الـ Referer فيه /admin."

يبقى أكيد المستخدم جه من Admin Panel.

يبقى اسمحله.

وده افتراض خاطئ جدًا.

ليه؟

لأن الـ Referer Header...

جاى منين؟

من الـ Client.

ومين يقدر يعدله؟

المستخدم.

يعني باستخدام Burp Suite.

أقدر أكتب.

```http
Referer:
/admin
```

حتى لو أنا أصلًا مفتحتش Admin Panel.

ولو التطبيق صدقني...

يبقى عندنا Broken Access Control.

# 💡 أهم الأسئلة اللي لازم تسألها لنفسك

كل ما تمسك Request...

اسأل نفسك:

### Authentication

- مين المستخدم الحالي؟
- هل لازم يكون Logged In؟

---

### Authorization

- مين المفروض يقدر ينفذ الـ Request دي؟
- هل فيه Role معين؟

---

### Access Control

- إيه اللي هيمنعني أجرب بنفس الـ Request؟
- هل ممكن أوصل لنفس الـ Function بطريقة تانية؟
- هل فيه خطوة ناقصة في التحقق؟

---

### Client-controlled Data

- هل فيه Parameter أقدر أعدله؟
- Cookie؟
- Header؟
- Hidden Field؟
- JSON Body؟

---

### Resource Identification

- التطبيق وصل للـ Resource دي إزاي؟
- User ID؟
- UUID؟
- File Name؟
- Order ID؟
- API Key؟
- Session؟

---

### HTTP

- هل تغيير الـ Method هيفرق؟
- هل تغيير الـ URL هيفرق؟
- هل فيه Rewrite Header؟
- هل فيه اختلاف في Routing؟

---

### Workflow

- هل العملية دي فيها أكتر من خطوة؟
- هل كل خطوة عليها Authorization؟
- ولا أول خطوة بس؟

---

### Impact

- لو الثغرة اشتغلت...
- إيه أسوأ حاجة ممكن أوصلها؟

# 🏁 الخلاصة

لو فيه فكرة واحدة بس أحبك تفتكرها بعد ما تخلص الملف ده كله...

فهتكون دي.

> **متدورش على الثغرة...**
>
> **دور على القرار الأمني اللي التطبيق أخده.**

كل Request في التطبيق...

فيه وراها قرار.

- هل اسمح؟
- ولا أمنع؟

وظيفتك كمختبر اختراق...

مش إنك تحفظ السيناريوهات.

وظيفتك إنك تكتشف:

**التطبيق اتخذ القرار ده بناءً على إيه؟**

ولو قدرت تعرف الإجابة...

هتعرف تختبر القرار.

ولو عرفت تختبر القرار...

هتعرف تكتشف ثغرات Broken Access Control حتى في تطبيقات عمرك ما شفتها قبل كده.
